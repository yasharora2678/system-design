SOLID Principles

Your Intent:

SOLID principles help us write clean, maintainable and scalable code. Instead of creating tightly coupled classes, SOLID guides us to design systems where change is easy, bugs are fewer, and logic is reusable.

Definition (From My Side)

SOLID is a set of five object-oriented design principles that improve code readability, extensibility, and maintainability. It helps developers design software that is easy to understand, modify, and scale.

S — Single Responsibility Principle

Meaning: A class should have only one reason to change.

class Invoice {
    void calculateTotal() {}
    void printInvoice() {}
    void saveToDB() {}
}

Better:

class InvoiceCalculator { void calculate() {} }
class InvoicePrinter { void print() {} }
class InvoiceRepository { void save() {} }

O — Open / Closed Principle

Meaning: Classes should be open for extension but closed for modification.

interface Discount {
    double apply(double price);
}

class FestivalDiscount implements Discount {
    public double apply(double price) {
        return price * 0.8;
    }
}

L — Liskov Substitution Principle

Meaning: Child classes should replace parent classes without breaking behavior.

class Bird {
    void fly() {}
}

class Sparrow extends Bird {}

Avoid forcing behavior that doesn't apply.

I — Interface Segregation Principle

Meaning: Do not force classes to implement unnecessary methods.

interface Worker {
    void work();
}

interface Eater {
    void eat();
}

D — Dependency Inversion Principle

Meaning: Depend on abstractions, not concrete classes.

interface Engine {
    void start();
}

class PetrolEngine implements Engine {
    public void start() {}
}

class Car {
    Engine engine;
    Car(Engine engine) {
        this.engine = engine;
    }
}

Why SOLID?

  • Reduces tight coupling
  • Improves readability
  • Makes systems scalable
  • Helps in LLD interviews
  • Supports clean architecture
